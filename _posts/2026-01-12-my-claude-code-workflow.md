<p><a href="https://code.claude.com/docs/en/overview">Claude Code</a> with Opus 4.5 is making waves. It really does feel like a step change in agentic coding tools &#8211; I built <strong><a href="http://flockwith.me">Flock</a></strong>, <strong><a href="https://medtrack.blmc.dev/">Medtracker</a></strong>, and <strong><a href="https://retreatsfyi.blmc.dev/">retreatsfyi</a></strong> without writing a single line of code (although I did do a bunch of debugging myself).</p><p>My tips below.</p><div><hr></div><h2>Pay for Max</h2><p>Max gets you much, much more usage, and is really the only feasible way to use Opus 4.5, which is a step change from older models. Just spend the money.</p><h2>Multiple instances</h2><p>Table stakes &#8211; you can run multiple instances of Claude Code to really rinse your usage limit. You can have Claude working on multiple features at the same time, or one main instance where you&#8217;re doing your work and then auxiliary instances that are doing other useful things like researching new features, merging in dependabot PRs, refactoring bits of the codebase etc.</p><p>Use worktrees as detailed below to make it really fly.</p><h2>My usual workflow</h2><ul><li><p>Check out a worktree branch</p></li><li><p>Open Claude in <code>--skip-permissions</code> mode (I am insane; you should do this in some sort of sandbox)</p></li><li><p>In <a href="https://code.claude.com/docs/en/common-workflows#use-plan-mode-for-safe-code-analysis">plan mode</a>, chat to Claude about what you want to build. This is especially good if you have a PRD somewhere in your repo which lists features you want to build and gives some project context.</p></li><li><p>When you&#8217;re happy, tell Claude to build your feature, making sure all tests pass and the code is linted (or add those hints to your CLAUDE.md).</p></li></ul><p>I&#8217;ll often have several Claude instances all writing up plans in multiple worktrees, and then review each plan as it comes in and get Claude to implement it.</p><h2>Worktrees and <a href="https://worktrunk.dev/">worktrunk</a></h2><p>Worktrees are super useful but have terrible ergonomics out of the box. You have to check out master, create the worktree in a specific directory, change dir, and then clean it all up when you&#8217;re done. I&#8217;m loving <a href="https://worktrunk.dev/">worktrunk</a>, which adds a nice layer and CLI on top of worktrees. My basic workflow:</p><ul><li><p><code>wt list</code> shows all your active worktrees</p></li><li><p>you can <code>wt switch --create &lt;branchname&gt; </code>to switch to a new worktree</p></li><li><p><code>wt merge</code> runs any hooks you have defined in your config file (like running unit tests, lint), merges the branch into <code>master</code>, and cleans it up</p></li></ul><p>You can also use a post-switch hook to launch a dev server on a specific port:</p><pre><code># Start dev server on a unique port per worktree (runs in background after switch)
post-switch = "bin/dev -p {{ branch | hash_port }}"</code></pre><p>It <a href="https://worktrunk.dev/claude-code/">integrates with Claude Code</a> to show you what branch you&#8217;re on and the URL to the local server for that branch in your statusline.</p><h2><a href="https://github.com/henryaj/autoclaude/">Autoclaude</a></h2><p>I <a href="https://github.com/henryaj/autoclaude/">wrote a tool</a> which looks at your current tmux window, identifies which panes are running Claude Code, and lets you watch for the &#8216;usage limit hit&#8217; notification. It then waits for your limit to reset and continues your session.</p><h2>&#8220;Do it for me&#8221;</h2><p>Wait&#8230; you mean you&#8217;re still doing things? You can just not do things!</p><p>Claude will often stop and give you instructions on how to proceed &#8211;&nbsp;checking app logs, checking the build status on GitHub, deploying an app to Heroku, configuring DNS on Route53. But it can do all those things itself!</p><p>Just ask Claude to &#8220;do it for me&#8221;. It&#8217;ll figure out what CLI tools you need, install them, and then do the thing for you (e.g. it&#8217;ll write a changeset to a tempfile and then pass it to the <code>aws</code> CLI to update your Route53 records).</p><h2>Ask Claude to interview you</h2><p>You can prompt Claude to use the AskUserQuestion tool to interview you. It&#8217;ll lie about how many questions you have left, so at some point you can tell it to stop and write up your answers. Claude already does this in plan mode, but by prompting it explicitly to use the tool you can use it to explore new features, vague ideas that aren&#8217;t yet fully-formed.</p><p>See <a href="https://x.com/trq212/status/2005315279455142243">this starter prompt</a>.</p><h2>Claude Code web</h2><p>I think it&#8217;s a bit of a meh product and there are <a href="https://www.terragonlabs.com/">better ones out there</a>. But you can ask it to bootstrap a setup script for your repo with some basic instructions (like telling it your stack and how it can boot a server, run lint and tests etc) and it&#8217;ll write one for you.</p><p>It&#8217;s still pretty good for oneshotting simple features, and it&#8217;s pretty fun to just smash out a few feature requests or bug fixes and they&#8217;re done by the time you get to your desk. You can use <a href="https://code.claude.com/docs/en/claude-code-on-the-web#from-web-to-terminal">teleport</a> to open the branch locally to test and make any finishing touches.</p><h2>Automated QA</h2><p>Install puppeteer and get Claude to open your running app in a browser and do its own integration testing. If it finds issues, it can fix them itself. Make sure to tell it to use headless mode if you don&#8217;t want random browser winders popping up all over the place.</p><h2>Claude with Chrome</h2><p>Start Claude with the <code>--chrome</code> flag to allow it to interface with the <a href="https://code.claude.com/docs/en/chrome">Claude for Chrome extension</a>. Then you can instruct Claude to use the browser to do anything you&#8217;d normally do. I recently used it to gather screenshots of <a href="https://flockwith.me">Flock</a> to update the homepage.</p><div><hr></div><p>I don&#8217;t think this is the last time we&#8217;ll see model and harness improvements that so drastically improve the way we write code and build products. The fact that there&#8217;s so much to get out of these relatively basic tips suggests that the harnesses could be improved much further.</p><p>Comment with any tips you have for getting the most out of Claude Code!</p><p></p>
